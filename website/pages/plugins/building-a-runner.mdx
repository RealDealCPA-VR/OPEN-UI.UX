# Building a Runner

<Callout type="info">
  Mirrored from [`docs/runner-authoring.md`](https://github.com/RealDealCPA-VR/opencodex/blob/main/docs/runner-authoring.md). See also the user-facing [Agent runners guide](../guides/runners/) and the [SDK API reference](./api/).
</Callout>

A "runner" in OpenCodex is anything that implements `SubagentRunner`. The built-in runner runs in-process through whichever `LLMProvider` you have configured; first-party adapters wrap Claude Code, OpenCode, and Aider; third parties ship their own through the plugin SDK.

## How runners relate to providers

An `LLMProvider` is the thinnest possible LLM client: one model, one streaming `chat()` call, one `embed()` call. A `SubagentRunner` sits one level up — it owns a whole agentic loop end-to-end: prompting, tool calling, file editing, and approval. The internal runner uses a provider under the hood; an external runner (Claude Code, OpenCode, Aider, your plugin) spawns a child CLI that brings its own provider, prompts, and tools.

In short: providers stream tokens; runners run tasks.

## The `SubagentRunner` interface

```ts
interface SubagentRunner {
  readonly id: string;          // unique across all runners
  readonly displayName: string; // shown in the picker
  readonly streaming: boolean;  // false → UI shows a spinner only
  checkInstalled(): Promise<{
    installed: boolean;
    version?: string;
    detail?: string;
    hintUrl?: string;
  }>;
  run(options: SubagentRunOptions): AsyncIterable<ChatEvent>;
}
```

`SubagentRunOptions` carries the prompt, the workspace `cwd`, the worktree path, any `allowedTools` advisory whitelist, and an `AbortSignal`.

## `ChatEvent` semantics

Whatever the underlying CLI streams (NDJSON, SSE, plain stdout), the agent loop only understands `ChatEvent`. The required terminal event is `done`. Conventionally, `usage` is emitted just before `done` so the cost meter has a final tally before the run is marked complete.

| Event | Semantics |
|---|---|
| `text_delta` | Incremental assistant text. Yield as bytes arrive — do not buffer the whole turn. |
| `tool_call` | A complete tool invocation: `{ id, name, input }`. `id` must be unique within the run. |
| `tool_result` | The CLI's observation of a tool call: `{ id, output, isError? }`. |
| `usage` | `{ inputTokens, outputTokens, cachedInputTokens? }`. Recommended once near the end. |
| `done` | **Required** terminal event. `stopReason` is one of `stop`, `tool_use`, `max_tokens`, `cancelled`, `runner_error`, `runner_not_installed`. |

<Callout type="warning">
  Don't throw out of the iterator on CLI failure — the loop translates a throw into a generic error and loses your streamed events. Yield `{ type: 'done', stopReason: 'runner_error', error: '<message>' }` instead.
</Callout>

If your CLI doesn't surface tool calls separately (Aider, for example, just writes diffs to disk), skip `tool_call` / `tool_result` entirely and emit one final `text_delta` summary plus `done`.

## `checkInstalled` contract

Called by Settings → Runners (to drive the status pill) and at the top of every `run()` (to short-circuit with `stopReason: 'runner_not_installed'` if the CLI vanished):

```ts
async checkInstalled() {
  try {
    const { stdout } = await execFile(this.cliPath, ['--version'], {
      timeout: 3000,
    });
    return { installed: true, version: stdout.trim() };
  } catch {
    return {
      installed: false,
      detail: `Not found on PATH. Override at Settings → Runners → ${this.id} → cliPath.`,
      hintUrl: 'https://example.com/install',
    };
  }
}
```

Fallback search paths: probe `PATH` first, then known per-platform install locations (`~/.local/bin`, `/opt/homebrew/bin`, `%APPDATA%\npm`), then the user-configured `cliPath` override. Keep the probe under a few seconds.

## Abort lifecycle

Every `run()` receives `opts.signal`. Wire it to the child process via the shared `treeKill` helper from `@opencodex/core`:

```ts
import { treeKill } from '@opencodex/core';

async function* run(options: SubagentRunOptions) {
  const child = spawn(this.cliPath, ['--prompt', options.prompt], {
    cwd: options.cwd,
    stdio: ['ignore', 'pipe', 'pipe'],
  });

  const cleanup = () => treeKill(child.pid).catch(() => {});
  options.signal.addEventListener('abort', cleanup, { once: true });

  try {
    for await (const event of parseStream(child.stdout)) {
      yield event;
    }
  } finally {
    cleanup();
  }
}
```

`treeKill` SIGTERMs the whole process tree — Aider, for instance, spawns its own `git` children, and they get cleaned up too.

## Plugin wiring

Manifest:

```json
{
  "name": "my-runner",
  "version": "0.0.0",
  "displayName": "My Runner",
  "entry": "dist/index.js",
  "engines": { "opencodex": "^0.1.0" },
  "permissions": ["agent.runner"],
  "contributions": {
    "runners": ["my-runner"]
  }
}
```

Entry point:

```ts
import { definePlugin } from '@opencodex/plugin-sdk';
import { myRunner } from './my-runner.js';

export default definePlugin({
  async activate(host) {
    host.registerRunner(myRunner);
  },
});
```

Without the `agent.runner` permission, `host.registerRunner` throws at activation.

## Worktree caveat

<Callout type="warning">
  External runners always run inside a git worktree. If the workspace is not a git repo, OpenCodex refuses to start the run and emits `done` with `stopReason: 'runner_error'`. There is no fallback.
</Callout>

OpenCodex creates a fresh worktree under `<workspace>/.opencodex/worktrees/<run-id>` on branch `opencodex/subagent/<run-id>`, sets it as the CLI's `cwd`, and queues the resulting diff for review when the run finishes. Design your adapter assuming `cwd` is always a throwaway branch.
