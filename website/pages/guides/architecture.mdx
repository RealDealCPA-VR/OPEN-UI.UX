## Goals

1. **Provider-agnostic.** Any LLM works. The provider layer is the most important abstraction in the codebase.
2. **Local-first.** No required backend. Your code, your keys, your machine.
3. **Extensible.** Third-party plugins and MCP servers extend the same registries that built-in code uses.
4. **Safe by default.** Every write or shell op goes through an explicit approval gate the user controls.

### Unified left column (Phase 10)

The renderer's chrome is a single left-edge surface that the **nav rail**
and the **context pane** share. The nav rail (icons for Chat, Files,
Agent, Automations, Plugins, Settings) is always 48px wide; the context
pane sits to its right and shows whatever is relevant to the current
view вҖ” conversation list, file tree, scheduled-task list, plugin list,
etc. Users can collapse the context pane to recover the pre-Phase-10
look; deep links from older sessions still resolve.

`Automations` is a top-level nav item (it used to live under
Settings вҶ’ Scheduled Tasks). Deep links to the old route 301 to the new
location.

The renderer is created with `sandbox: true`, `contextIsolation: true`, `nodeIntegration: false`. Every privileged operation crosses the typed IPC bridge.

## Provider abstraction

Every LLM goes through one interface:

```ts
interface LLMProvider {
  readonly id: string;
  readonly displayName: string;

  chat(req: ChatRequest): AsyncIterable<ChatEvent>;
  embed(req: EmbedRequest): Promise<EmbedResult>;
  listModels(): Promise<ModelCapabilities[]>;
  capabilities(model: string): Promise<ModelCapabilities | undefined>;
}
```

Providers are constructed by a `ProviderFactory` that owns a Zod `configSchema`. The `ProviderRegistry` validates config at create-time and throws `ProviderConfigError` on schema failure.

A `ChatEvent` is a discriminated union of `text_delta`, `tool_call`, `tool_result`, `usage`, `done`, `error`. Provider adapters translate their native streaming format into this common event series.

## Tool layer

A `Tool` extends a `ToolDefinition` with an `inputZod` schema and an `execute(input, ctx)` function. Each declares a `permissionTier`: `'read' | 'write' | 'execute' | 'network'`.

Tools come from three sources, all flowing through the same registry:

1. **Built-in** (`@opencodex/tools`) вҖ” `read_file`, `write_file`, `edit_file`, `glob`, `grep`, `list_dir`, `run_shell`, `web_fetch`, `search_codebase`, `spawn_subagent`.
2. **MCP** (`@opencodex/mcp-client`) вҖ” every connected MCP server's tools.
3. **Plugins** (`@opencodex/plugin-sdk`) вҖ” third-party tools registered via the plugin host.

## Agent loop

`startChatStream()` appends the user message, creates an empty assistant message, expands stored history into provider-shaped `Message[]`, then dispatches `runStream()`.

`runStream()` iterates over `provider.chat({ model, messages, tools, signal })`, accumulating `text_delta` into a buffer and collecting `tool_call` events. After each provider turn:

- If `stopReason === 'tool_use'` and tools were called, each tool runs through `executeToolCall()` вҖ” which checks the approval policy, executes via the `ToolRegistry`, audits the call, and pushes a `tool_result` block back into history for the next iteration.
- Otherwise the loop emits `done` and exits.

Hard cap at `MAX_TOOL_ITERATIONS = 10` prevents runaway tool loops.

## IPC contract

The wire format is typed in `apps/desktop/src/shared/ipc-types.ts`. Every invoke handler is registered through `registerInvoke()`, which `safeParse()`s the raw payload on every call against a Zod schema.

## Storage

- **SQLite** via `better-sqlite3` in the OS user-data dir, WAL mode, `foreign_keys = ON`. Schema is versioned through migrations.
- **`electron-store`** for non-secret preferences, Zod-validated.
- **`keytar`** for secrets вҖ” provider API keys live in the OS keychain under service `opencodex`. Never logged, never in SQLite or settings JSON.

## RAG / codebase chat

- **Chunker**: tree-sitter for AST-aware splits.
- **Storage**: LanceDB for vectors + SQLite FTS5 for keyword.
- **Retrieval**: hybrid via Reciprocal Rank Fusion.
- **Embeddings**: provider-agnostic (same `LLMProvider.embed` interface).
- **Updates**: chokidar file watcher вҶ’ incremental reindex.

## Multi-agent orchestration

- `spawn_subagent` tool lets an orchestrator agent fan out work. Each spawn picks a `runnerId` from the `SubagentRunner` registry.
- The **internal** runner runs in an Electron `utilityProcess` with its own provider, context, and tool subset. `ChatEvent`s stream back through the IPC bridge in real time, and the host's approval gateway fires for every write/execute tool call.
- **External runners** вҖ” `@opencodex/runner-claude-code`, `@opencodex/runner-opencode`, `@opencodex/runner-aider`, plus any plugin-registered runner gated by the `agent.runner` permission вҖ” spawn the corresponding CLI as a child process. Their tool calls happen out-of-process; OpenCodex's per-call approval modals do not fire, and the CLI's own approval system is authoritative.
- Every external run is **worktree-only**: OpenCodex creates a fresh worktree under `<workspace>/.opencodex/worktrees/<id>` on branch `opencodex/subagent/<id>`, sets it as the CLI's cwd, and queues the resulting diff for merge-review. Non-git workspaces refuse to start an external run.
- Cancellation goes through the shared `treeKill` helper so spawned grandchildren (e.g. a `git` invocation an external runner started) get cleaned up too.
- Budget caps prevent runaway spawning.

See the [Agent runners guide](./runners/) for the user-facing perspective and the [SDK API reference](../plugins/api/#runner-sdk) for the `SubagentRunner` shape.

## MCP integration

When a server connects, its `tools/list` is registered through the same `ToolRegistry`, its `resources/list` feeds the RAG retriever, and its `prompts/list` shows up as `/` commands in the chat.

## Plugin model

Plugin packages ship a `opencodex.plugin.json` manifest declaring `permissions[]` and `contributions{}`. On install the user reviews permissions and confirms. The loader hands the entry module a `PluginHost` restricted to declared permissions, and registers contributions with the relevant registries.

## Why Electron, not Tauri

Electron chosen for iteration velocity. Pure Node/TS means the same SDKs work everywhere without a RustвҶ”TS boundary. Cost: bundle size (~150MB) and a slightly weaker sandbox model вҖ” both mitigated by aggressively using `sandbox: true` + `contextIsolation: true` and routing all privileged work through the IPC bridge.
