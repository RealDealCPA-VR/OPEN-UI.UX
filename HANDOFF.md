## Context Notes

### Re-run data flow

- **`onRerun` flows top-down**: `ChatPane.handleRerun(prompt: string)` → passed once to each `MessageBubble` / `DraftBubble` → forwarded to `BlockSequence` → handed to every `ToolCallCard` in that bubble.
- **`handleRerun` does three things**: `setInput(prompt)`, `inputRef.current?.focus()`, then `setSelectionRange(len, len)` to put the caret at the end. Composer is a controlled `<textarea>` so the value flips on the next render — the `setSelectionRange` only needs to fire after focus, both happen synchronously inside the click handler.
- **`inputRef`** is a `useRef<HTMLTextAreaElement | null>` on `ChatPane`. Attached via `ref={inputRef}` on the composer textarea. Don't remove it — the focus behavior depends on it.
- **Prompt format** is `Re-run this tool call: <name>(<args>)`. Args are pretty-printed JSON (2-space indent) via the existing `formatToolArguments` helper. Null/undefined args collapse to `name()`. **Watch out**: very long args (e.g. a multi-kb `write_file` content) will produce a multi-line textarea — that's fine, the textarea grows with its `rows={3}` default but the user can scroll inside it.

### Why Re-run lives in the head, not the body

- Earlier instinct was "put Re-run in the body alongside Arguments / Result" — but cards default to **collapsed**, so the user would have to expand first. Putting it in the head keeps it one-click and makes "I want to re-run this exact call" the fast path. The trade-off is the head got slightly busier; the layout still has plenty of room thanks to `min-width: 0` on the toggle.
- **The head is no longer a single `<button>`.** It's a `<div class="tool-card-head">` containing `<button class="tool-card-head-toggle">…</button>`, the pill `<span>`, and `<button class="tool-card-rerun">Re-run</button>`. Nesting buttons is invalid HTML — if you add more controls (e.g. "Copy as cURL", "Open in tab"), keep them as siblings.

### Disabled-while-pending decision

- `disabled={status === 'pending'}` reflects "result hasn't arrived yet". Re-running mid-flight would create a confusing parallel call. If the user really wants to re-issue while waiting they can copy from the Args panel.
- **Approval-modal-open state isn't distinguished from "tool executing"** — both are `result === null`. Same caveat as carrying over: if you want to surface "waiting on user approval" separately, the runner / `ApprovalManager` needs to emit a new status event.

### Carry-overs still relevant (all from previous sessions)

- **Block ordering is canonical for both persisted and live.** Persisted: runner's `allBlocks` accumulates `[text1, tool_use1, tool_result1, text2, tool_use2, tool_result2, ..., final_text]` and writes it to `content_blocks_json` (see [`runner.ts`](apps/desktop/src/main/chat/runner.ts) — `runStream` finally). Live: the chat-context reducer mirrors this — `text_delta` extends the trailing text block, `tool_call` appends a `tool_use` block, `tool_result` appends a `tool_result` block.
- **Pairing semantics.** [`groupContentBlocks`](apps/desktop/src/renderer/components/tool-block-grouping.ts) walks blocks in order, indexes each `tool_use` by id, and when a `tool_result` arrives it patches the matching grouped item's `result` field. If a `tool_result` arrives without a preceding `tool_use` (shouldn't happen but defensive), it gets a synthetic `tool_use` with `name: 'unknown'` so the renderer still has something to draw.
- **`text` is still kept on `AssistantDraft`** alongside `blocks` — backwards compat with the export flow and any UI that wants a flat string. Always set both when extending the reducer.
- **`MessageBubble` decides between blocks vs. plain content.** `hasBlocks = message.contentBlocks !== null && message.contentBlocks.length > 0`. For old text-only persisted rows, `contentBlocks` is null (by design — runner only persists blocks when the turn includes tool blocks) so the bubble renders `message.content` through `Markdown` as before.
- **Pending state is fuzzy.** `result === null` ⇒ pill says `Running`. This covers both "approval modal is open" and "tool is actually executing" — they look identical. If you want to distinguish them you'd need the runner to emit a new event when approval is granted (or `ApprovalManager` to broadcast a status), then surface a third state.
- **Output rendering is generic JSON-stringify.** [`formatToolOutput`](apps/desktop/src/renderer/components/tool-block-grouping.ts) returns strings as-is and JSON-stringifies everything else with 2-space indent. This is the natural place to add per-tool views (task #4 above) — branch on `use.name` in `ToolCallCard` before falling through to `formatToolOutput`.
- **`ApprovalManager`** ([`approvals.ts`](apps/desktop/src/main/chat/approvals.ts)) takes `broadcaster`, `policyReader`, `policyWriter` in its constructor — **don't import `settings.ts` from `approvals.ts`** (`new Store(...)` at module load fails outside Electron and breaks `runner.test.ts`). [`approval-handlers.ts`](apps/desktop/src/main/chat/approval-handlers.ts) wires the real settings reader/writer in production.
- **Session overrides keyed by `streamId`**, not conversationId; cleared in `runStream` finally via `approvals?.clearSession(streamId)` which also rejects pending requests.
- **`contentBlocks` is null for text-only turns** by design — backwards compat. Only persisted when `allBlocks.some(b => b.type === 'tool_use' || b.type === 'tool_result')` ([`runner.ts`](apps/desktop/src/main/chat/runner.ts) finally).
- **The pre-created assistant placeholder row** (empty content, null blocks) is filtered out of LLM history via `m.id !== assistantRow.id` in `runStream`. Don't remove this filter.
- **Run-shell process-tree kill**: on Windows, `child.kill()` only kills `cmd.exe`; orphan node grandchild keeps pipes open and `close` never fires. [`tryKill()`](packages/tools/src/run-shell.ts) spawns `taskkill /F /T /PID <pid>` on Windows, uses `process.kill(-pid, 'SIGTERM')` on POSIX (requires `detached: true` at spawn).
- **`OPENCODEX_NO_RIPGREP=1`** forces the JS grep path (used in [`grep.test.ts`](packages/tools/src/grep.test.ts) for determinism). Ripgrep availability is cached in a module-level promise; `resetRipgrepAvailabilityCache()` clears it for tests.
- **Web-fetch allowlist** = `OPENCODEX_WEB_FETCH_ALLOWLIST` (lowercased, comma-separated). Empty = deny all. `*.example.com` matches `example.com`, `api.example.com`, etc. Redirects follow by default — allowlist gates the **initial** URL only. Future hardening: `redirect: 'manual'` + walk hops.
- **`atomicWrite`** ([`atomic-write.ts`](packages/tools/src/atomic-write.ts)) is the single safe-write chokepoint; new tools should not invent their own write path.
- **`edit_file` replacement is literal**, not regex — `EditFileAmbiguousError` is by design.
- **`zodToJSONSchema` supported subset**: `string` (url/email/uuid), `number`, `boolean`, `enum`, `array`, `object`, `optional`, `default`, `nullable`, `.describe()`, `record`, `additionalProperties: false`. `z.union()` / `z.date()` still require extending the converter.
- **`resolveWithinWorkspace`** ([`path-guard.ts`](packages/tools/src/path-guard.ts)) is the single path-escape chokepoint. `run_shell` uses it for `cwd`.
- **Folder path has a space + period** (`OPEN UI.UX`) — quote in shell.
- **`packages/*` tsconfig is `noEmit: true`** — before any `pnpm publish`, swap to `tsup` per package.
- **`.github/workflows/ci.yml`** still doesn't run `pnpm build`.
