# UI/UX Consistency Review — non-Settings UI

**73 findings — 10 high, 35 medium, 28 low.**

_Generated from a multi-agent review of `apps/desktop/src/renderer` (5 lenses). The Settings UI was fixed separately; this is the remaining backlog for the rest of the app._

## color-tokens

The renderer's non-settings UI is broadly token-driven, but two systemic color-token problems recur. (1) Pervasive `var(--token, #fallback)` inline styles whose hex/rgba fallback is a DIFFERENT, often light-theme or generic Tailwind-ish color that does not match the real dark-theme token value (e.g. `var(--danger, #dc2626)` while `--danger` is `#f87171`; `var(--success, #22c55e)` while `--success` is `#4ade80`; replay/diff cards fall back to light `#fee`/`#fcc`/`#900`). Per FIX RULE 2 these redundant/stale fallbacks should be dropped to just `var(--token)`, because if the var ever fails to resolve the fallback paints the wrong color and breaks theming. (2) Two modal scrims hardcode `rgba(0,0,0,0.4)`/`rgba(0,0,0,0.5)` instead of the existing `--shadow-overlay` token (one even has className="modal-overlay" which already sets it, then overrides inline). A handful of one-off shadows hardcode `rgba(0,0,0,...)` with no shadow token. Intentional literals (theme-preview gradients in ThemePanel, the xterm terminal background) are listed as non-issues and left alone. Note: many of the matched lines live in views/\*Panel.tsx Settings files which are out of scope for this pass and excluded below.

## layout-spacing

The non-settings UI has two recurring layout-spacing divergences. (1) Several modals hand-roll their own fixed/overlay backdrop + panel card with inline styles instead of using the shared Modal component or the existing .approval-modal-backdrop/.approval-modal classes. DraftPrModal does it correctly via <Modal>, but ReplayConversationModal and FanoutConsentModal duplicate the scaffold (and FanoutConsentModal even invents .modal-overlay/.modal-panel class names that don't exist in styles.css), losing the shared focus-trap/Escape/inert behavior. (2) Pervasive var(--token, #hexFallback) redundant fallbacks, plus two phantom tokens never defined in :root — --radius-md and --shadow-popover — which ALWAYS fall through to their inline fallback, so those surfaces never theme. The real radius token is --radius (9px) and the popover shadow standard is --shadow-dropdown. Raw gap/padding numbers are mostly on the 4/8 scale, so the higher-value fixes are de-duplicating the modal scaffolding and replacing phantom tokens / redundant fallbacks with the real tokens. Note: I report --radius-md/--shadow-popover usages as fixable by remapping to existing tokens (--radius / --shadow-dropdown) rather than as missing CSS, since equivalent real tokens already exist. styles.css was not edited (shared file).

## component-reuse

The non-settings UI has three systemic component-reuse divergences from the design system. (1) Several dialogs (FanoutConsentModal, ReplayConversationModal) hand-roll the entire modal scaffold — raw fixed overlay with hardcoded rgba(0,0,0,0.x) scrims, inline panel boxes, and manual role/aria-modal wiring — instead of the shared <Modal> component used correctly by AgentSpawnModal/DraftPrModal. (2) Buttons and inputs are frequently bespoke: unclassed <button> elements (Cancel/Change in AgentSpawnModal), a danger action styled with inline border/color instead of .btn-danger (FanoutConsentModal), pill chips with borderRadius:999 literals instead of .btn or var(--radius-pill), and text inputs/textarea styled inline instead of .settings-input (DraftPrModal, FanoutConsentModal, ChatBudgetOverride, ReplayConversationModal). The Modal rewrites are the highest-value fixes; the fallback cleanup is the broadest. None require styles.css edits except confirming no new shared class is needed.

## a11y-semantics

The non-settings renderer UI is broadly strong on accessibility: icon-only close/dismiss buttons consistently carry aria-labels, status pills (ToolCallCard, AgentRunRow, ActiveRunCard, StatusBar, Toasts) pair color with text labels so nothing is color-only, most form inputs are wrapped in associated <label> elements, AgentView/CommandPalette use correct tab/listbox/option roles, and AppShell ships a skip link plus labelled landmarks. The real gaps are: a role="button" article that nests interactive children (ActiveRunCard), two dialogs missing an accessible name (MergeReviewModal, ScheduledTaskEditorModal), three text inputs that rely on placeholder-only labelling (ChatView composer, CodebaseSearchBox, FileTree annotation via title), and a combobox/listbox in CommandPalette that doesn't expose the active option to screen readers. No color-only status signals were found.

- **[HIGH]** `components/ActiveRunCard.tsx:55` — The card root <article role="button" tabIndex={0} onClick={onSelect}> contains an <h3> title AND real interactive <button> elements (Open, Abort, Confirm/Cancel). Nesting focusable/interactive controls inside an element with role="button" is invalid ARIA and produces a confusing screen-reader tree (a button inside a button). It also has no aria-label, so the whole card is announced only by its concatenated inner text.
  - _Fix:_ Don't make the whole card a button. Drop role="button"/tabIndex/onKeyDown from the <article>, and instead make the run title (or an explicit Open button) the activating control — the 'Open' button at line 182 already calls onSelect(). If a whole-card click affordance must stay, keep onClick on the article for mouse convenience but remove the role/tabIndex so it is not exposed as a single button wrapping other buttons, and rely on the inner buttons for keyboard/AT access.
- **[HIGH]** `components/MergeReviewModal.tsx:470` — <div className="approval-modal-backdrop" role="dialog" aria-modal="true"> has no accessible name (no aria-label or aria-labelledby). The <h2>Review subagent changes</h2> at line 477 is not referenced. Screen readers announce an unnamed dialog. Peer modals (AgentSpawnModal, ReplayConversationModal, Modal) all set aria-labelledby.
  - _Fix:_ Give the <h2> an id (e.g. id="merge-review-modal-title") and add aria-labelledby="merge-review-modal-title" to the role="dialog" element, matching the AgentSpawnModal pattern (aria-labelledby="agent-spawn-modal-title").

  - _Fix:_ Add id="scheduled-task-editor-title" to the <h2> and aria-labelledby="scheduled-task-editor-title" to the role="dialog" div, consistent with the other modals in the codebase.
- **[MEDIUM]** `views/ChatView.tsx:883` — The primary chat composer <textarea className="chat-input"> has no associated <label> and no aria-label — only a dynamic placeholder (composerPlaceholder(...)) that changes with model/streaming state. ## typography-hierarchy

Reviewed heading hierarchy, inline font sizing, and empty/loading/error-state styling across the non-settings renderer UI (views/_, components/_). The design system standard is: page titles use the CSS-driven `.view h1` (24px / weight 600), sub-sections use `.settings-section-head` or scoped `h2`/`h3` selectors, and panels never hand-size headings inline. The biggest real divergences: (1) ReviewView hand-sizes its page `<h1>` to 16px inside an `<aside>` instead of using the `.view h1` page-title convention every other view (AgentView, CodebaseView, AutomationsView, RunnersView) relies on; (2) loading and empty states are conflated under the same bespoke classes with no consistency between views — ChatView's loading uses `chat-empty` while AgentView/AutomationsView loading uses `audit-empty`; (3) modal/error headings are inconsistently sized — some `<h2>` are styled by a scoped wrapper class (AgentSpawnModal, DraftPrModal, MergeReviewModal, AgentRunDrawer), while FanoutConsentModal and ErrorBoundary hand-size `<h2>` to 16px/14px inline. Color/radius fallback tokens (var(--x,#hex)) are pervasive but out of this lens; flagged only where they coincide with a heading. No styles.css edits proposed (shared file). Findings are read-only; no edits made.
